# Filesystem Store

**Date:** 2026-03-10
**Status:** Draft
**Depends on:** `store/` package interfaces, `encoding/xdbjson` package

## Overview

Implement `store/fsstore`, a filesystem-backed `store.Store` that persists records and schemas as JSON files. Uses the directory hierarchy to represent the namespace/schema/id structure and the `encoding/xdbjson` package for record serialization.

This store is suitable for local development, CLI tools, configuration storage, and any scenario where a simple, human-readable persistence layer is needed without running a database.

## Goals

- Implement every method in `store.Store` (records, schemas, namespaces)
- Implement `store.HealthChecker`
- Thread-safe via `sync.RWMutex` for concurrent in-process access
- Persist all data as JSON files on the local filesystem
- Human-readable storage format ã files can be inspected and edited by hand
- Port `encoding/xdbjson` from `xdbx` to `xdb` as a prerequisite
- Full test coverage using TDD

## Non-Goals

- `store.BatchExecutor` ã filesystem operations are not transactional; defer to a future plan if needed
- Multi-process concurrent access ã file locking across processes is out of scope
- File watching or change detection

## Directory Layout on Disk

The fsstore maps the XDB data model to directories and files:

```
<root>/
  <namespace>/
    <schema>/
      _schema.json          # Schema definition
      <id>.json             # Record data
```

### Examples

```
root/
  myapp/
    users/
      _schema.json          # schema.Def for "users"
      user-1.json           # Record: myapp/users/user-1
      user-2.json           # Record: myapp/users/user-2
    posts/
      _schema.json          # schema.Def for "posts"
      post-abc.json         # Record: myapp/posts/post-abc
  other-ns/
    config/
      _schema.json
      main.json
```

### Design Rationale

- **Directories = namespaces + schemas** ã natural mapping, easy to browse
- **`_schema.json`** ã underscore prefix avoids collision with record IDs
- **One file per record** ã simple reads/writes, git-friendly diffs
- **JSON format** ã human-readable, editable, uses existing `xdbjson` encoder/decoder

## Data Model

```go
type Store struct {
    mu   sync.RWMutex     // Protects all filesystem operations
    root string           // Root directory for all data
    enc  *xdbjson.Encoder // Record ã JSON
    opts Options
}

type Options struct {
    // Indent controls JSON pretty-printing.
    // Empty string means compact JSON.
    // Default: "  " (two spaces).
    Indent string
}
```

The `sync.RWMutex` serializes all write operations and allows concurrent reads. This follows the same pattern as `memstore` ã read methods acquire `mu.RLock()`, write methods acquire `mu.Lock()`. The lock protects against races in check-then-act sequences (e.g., `CreateRecord` checks file doesn't exist then writes it).

No in-memory state ã all reads go to disk, all writes go to disk. The filesystem is the source of truth.

## Path Mapping

Helpers in `paths.go` convert between URIs and filesystem paths:

```go
// recordPath returns the file path for a record.
// e.g., root/myapp/users/user-1.json
func (s *Store) recordPath(uri *core.URI) string

// schemaPath returns the file path for a schema definition.
// e.g., root/myapp/users/_schema.json
func (s *Store) schemaPath(uri *core.URI) string

// nsDir returns the directory for a namespace.
// e.g., root/myapp/
func (s *Store) nsDir(uri *core.URI) string

// schemaDir returns the directory for a schema.
// e.g., root/myapp/users/
func (s *Store) schemaDir(uri *core.URI) string
```

3. **`schemas.go`** ã Implement `SchemaReader` and `SchemaWriter`:

   - `UpdateSchema` ã overwrite `_schema.json`, error if not exists

### Phase 4: Record Operations

4. **`records.go`** ã Implement `RecordReader` and `RecordWriter`:
   - `CreateRecord` ã encode record with `xdbjson.Encoder`, write `<id>.json`, error if exists
   - `GetRecord` ã read `<id>.json`, decode with `xdbjson.Decoder`, return `ErrNotFound` if missing
   - `UpdateRecord` ã overwrite `<id>.json`, error if not exists
   - `UpsertRecord` ã write `<id>.json` unconditionally (create dirs if needed)
   - `DeleteRecord` ã remove `<id>.json`, error if not exists
   - `ListRecords` ã list `*.json` files (excluding `_schema.json`), decode, paginate

### Phase 5: Namespace Operations

5. **`namespaces.go`** ã Implement `NamespaceReader`:
   - `ListNamespaces` ã list top-level directories under root that contain at least one `_schema.json`, paginate
   - `GetNamespace` ã check if namespace dir exists with any `_schema.json`, return `ErrNotFound` if not

## File I/O Conventions

- **Atomic writes** ã write to a temp file in the same directory, then `os.Rename` to target path. This prevents partial writes on crash.
- **Directory creation** ã use `os.MkdirAll` when creating records/schemas to ensure parent dirs exist.
- **File permissions** ã directories: `0755`, files: `0644`.
- **Schema file name** ã `_schema.json` (underscore prefix reserved, record IDs must not start with `_`).

## Key Design Decisions

| Decision                          | Rationale                                                                                       |
| --------------------------------- | ----------------------------------------------------------------------------------------------- |
| No in-memory cache                | Simple, no stale data. Performance is acceptable for local/CLI use                              |
| Atomic writes via temp + rename   | Prevents corrupted files on crash                                                               |
| `_schema.json` for schema defs    | Colocated with records, underscore avoids ID collision                                          |
| Directories for ns/schema         | Natural mapping, browsable, git-friendly                                                        |
| `sync.RWMutex` for concurrency    | Same pattern as memstore; serializes writes, allows concurrent reads within a single process    |
| No `BatchExecutor`                | Filesystem ops aren't transactional; keep it simple                                             |
| JSON pretty-print by default      | Human-readable files; compact mode available via `Options`                                      |
| Namespaces derived from dirs      | Consistent with store interface having no `NamespaceWriter`                                     |
| `xdbjson` for serialization       | Reuse existing encoder/decoder; handles type conversion, nested attrs, metadata fields          |

## Test Plan

Tests follow TDD ã write each test before the implementation. Use `t.TempDir()` for isolated test directories.

### Shared Test Suites

Run all shared test suites from `tests/` against `fsstore.Store`:
- `RecordStoreSuite`
- `SchemaStoreSuite`
- `NamespaceStoreSuite`

### FS-Specific Tests

#### File Layout

- Create a schema, verify `_schema.json` exists at expected path
- Create a record, verify `<id>.json` exists at expected path
- Delete last schema in namespace, verify directory is cleaned up

#### Atomic Writes

- Verify records are written atomically (no partial files)

#### Path Mapping

- URI with simple ns/schema/id maps to correct path
- IDs with special characters are handled correctly

#### Health Check

- `Health()` succeeds with valid root directory
- `Health()` fails with non-existent root directory
- `Health()` fails with non-writable root directory

#### Concurrency

- Concurrent reads don't block each other
- Concurrent writes are serialized correctly
- Read during write sees consistent state
- Concurrent creates of the same record ã one succeeds, one returns `ErrAlreadyExists`

#### Edge Cases

- Record ID starting with `_` ã should be rejected or handled to avoid `_schema.json` collision
- Empty namespace/schema directories are cleaned up on delete
- Very long file paths ã OS limits
